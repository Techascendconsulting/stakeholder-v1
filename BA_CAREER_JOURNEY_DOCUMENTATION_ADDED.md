### Real-World Context
"You've gathered requirements from stakeholders. Now you need to document them clearly so developers, testers, and designers can understand exactly what needs to be built. In Agile, this means writing user stories with acceptance criteria. In Waterfall, it means detailed requirements specifications."

### Topics (6 activities):

1. **Writing User Stories** (1 hour)
   - Write clear user stories following "As a... I want... So that..." format
   - Links to: documentation view

2. **Defining Acceptance Criteria** (1 hour)
   - Write testable acceptance criteria that define "done"
   - Links to: documentation view

3. **Functional Requirements Documentation** (2 hours)
   - Document what the system must do using clear, unambiguous language
   - Links to: requirements-engineering view

4. **Non-Functional Requirements** (1 hour)
   - Capture performance, security, usability, and compliance requirements
   - Links to: requirements-engineering view

5. **Requirements Validation** (45 min)
   - Review documentation with stakeholders to ensure accuracy and completeness
   - Links to: training-practice (stakeholder meeting)

6. **Requirements Traceability** (30 min)
   - Link requirements back to business objectives and forward to test cases

**Total Time**: ~6 hours 15 minutes

### Key Deliverables
- User stories with acceptance criteria
- Functional requirements document
- Non-functional requirements specification
- Requirements traceability matrix
- Stakeholder sign-off on requirements
